# cub3D ガイド分かりやすさの分析

このガイドが「超初心者でも分かる」と評価された理由を分析し、
他のガイド（CPP Modules 等）にも展開できる**再現可能な原則**として整理した。

---

## 📊 改善の変遷（Before → After）

### Before（初期版）
- 6 ページ × 平均 580 行の長大ページ
- ASCII アート図（ブラウザで崩れる）
- 専門用語を説明なしで使用（DDA、camera_x、struct 等）
- 英語カタカナ（レイキャスティング、DDA）のみ
- 図は少なく文章ベース
- プログラム本体（main.c）の解説なし

### After（改善版）
- 11 ページに分割、1 ページ 10〜15 分で読める量
- Mermaid 図 + 表形式で視覚化
- 用語集ページを新設、初出時に平易な定義
- 英語 + 日本語訳併記（光線投射法、デジタル微分解析機）
- 教育用 GIF 4 本 + 実ゲームスクリーンショット
- main.c / init.c / render.c の「プログラム全体」ページ追加

## ✅ 効いた 10 の原則

### 原則 1: 「このページは何？」を最初に 1 行で

```markdown
## このページは何？

**光線が壁を見つけるまで、地図をどう進むかを解説するページ** です。
```

→ 読者は **5 秒で** そのページを読むべきか判断できる。

### 原則 2: 身近な例え（メタファー）から始める

- レイキャスティング → **懐中電灯**
- 構造体 → **引き出し付きの整理箱**
- private メンバ → **鍵付きの日記帳**

→ 専門知識がなくても **直感的に** 概念が入る。

### 原則 3: 英語 + 日本語併記

```markdown
- **レイキャスティング (raycasting)** = **光線投射法**
- **DDA** = **Digital Differential Analyzer** = **デジタル微分解析機**
```

→ **カタカナだけ** で押し通さず、日本語で意味を伝える。

### 原則 4: 「〇〇って何？」サブセクションで 1 つずつ解説

```markdown
### 光線（ray）って何？
### DDA って何？
### カメラプレーンって何？
```

→ 読者が **知らない概念を 1 つずつ** 片付けられる。
専門用語を使う前に必ず定義。

### 原則 5: ASCII アートは極力使わず、Mermaid・表・GIF に置き換える

ASCII アートはブラウザで等幅フォント崩れ・フォント差で歪む。

- プロセスフロー → **Mermaid flowchart**
- 比較対応表 → **Markdown テーブル**
- 動作イメージ → **GIF（Python 生成 or 実機録画）**

### 原則 6: 数値や変数名を具体的に示す

❌ 「画面幅分の光線を飛ばす」
✅ 「画面幅 1024 ピクセルだから **1024 本** の光線を飛ばす」

❌ 「プレイヤー位置を小数で保持」
✅ 「マスの中央に置くため **0.5 足して** (3.5, 2.5) にする」

→ **なぜその値か** まで説明すると、読者が納得できる。

### 原則 7: コードは 1 行 1 行コメント

```c
// ── ① miniLibX 起動 ──
// X11 や OpenGL との接続を確立
game->mlx = mlx_init();
if (!game->mlx)
    ft_error("Failed to initialize miniLibX");
```

→ **何をしている行か** が全行分かる。読み飛ばしても構造が見える。

### 原則 8: ページを短く保ち、番号で分割

- 03-raycasting（概念）
- 04-dda（アルゴリズム）
- 05-camera（補正）

→ 「1 ページ 30 分」より「3 ページ × 10 分」の方が**集中が続く**。

### 原則 9: エラーパターン・よくあるミスを具体的に

```markdown
=== "🔴 NG: 開いた壁"
    1 1 1 1 1
    1 0 0 0 1
    1 0 N 0 1
    1 1 0 1 1   ← 下が '0' で開いている
```

→ **正常例だけでなく異常例** を並べると、読者は判断基準を身につけられる。

### 原則 10: 用語集ページと相互リンク

- 各ページ冒頭に「📚 用語集」リンク
- 用語集から主要ページに戻るリンク

→ 「分からない単語があったらそこに行ける」**安心感** を提供。

## 🎨 デザインの原則

### 色の使い方（教科書風）

| 色 | 意図 | Mermaid で使う場面 |
|:-:|:---|:---|
| 🟦 水色 (`#E3F2FD`) | スタート・入口 | プロセスの始点 |
| 🟨 黄色 (`#FFF9C4`) | 要検討・判定 | 条件分岐 |
| 🟩 緑 (`#C8E6C9`) | 成功・完了 | 処理の終点 |
| 🟧 オレンジ (`#FFE0B2`) | 初期化・準備 | セットアップ |
| 🟥 赤 (`#FFCDD2`) | エラー・警告 | 失敗パターン |
| 🟪 紫 (`#F8BBD0`) | ループ・反復 | 繰り返し処理 |

→ **3〜5 色に抑える**。増やしすぎると逆に分かりにくい。

### 視覚要素の使い分け

| 要素 | 用途 |
|:---|:---|
| `!!! info` | 補足情報、「なぜ〇〇するのか」 |
| `!!! tip` | 便利なコツ、ディフェンス対策 |
| `!!! warning` | よくあるミス、注意点 |
| `!!! danger` | 即不合格フラグ、重大な禁止事項 |
| `!!! note` | 評価シート原文引用 |
| タブ (`=== "..."`)  | C と C++ の比較 |
| テーブル | 対応関係、判定基準 |
| Mermaid | プロセス・フロー・関係図 |
| GIF / PNG | 動作・見た目のイメージ |

## 🏗️ 構成の原則

### ページテンプレート（12 セクション）

1. **このページは何？** - 平易な言葉で 2〜3 行
2. **このページで学ぶこと** - 箇条書き
3. **新しい概念の解説** - 「〇〇って何？」で 1 つずつ
4. **具体例** - 数字付きで
5. **C との比較** - タブ形式
6. **コード解説** - 1 行 1 行コメント
7. **評価シート原文** - 引用ボックス
8. **テストチェックリスト**
9. **ディフェンス Q&A** - 質問/答えテーブル
10. **よくあるミス** - Warning ボックス
11. **用語集へのリンク**
12. **次のページへ**

### サイト全体構成

```
ホーム (目次)
  ↓
プログラム全体の流れ (入口)
  ↓
各モジュール (番号順)
  ↓
評価対策 (ディフェンス準備)
  ↓
用語集 (いつでも参照)
```

## 🚫 避けるべきこと

### アンチパターン

| NG | なぜダメ |
|:---|:---|
| ASCII アート | ブラウザで崩れる、等幅フォント依存 |
| 専門用語の説明なし | 読者がつまずく |
| 1 ページ 500 行超え | 集中が切れる |
| カタカナのみ | 意味が入らない |
| 色を 10 種類以上使う | 視覚的に騒がしい |
| コメントなしコード | 何をしているか不明 |
| 抽象的な説明のみ | イメージできない |
| Permalink ¶ マーク | 絵文字と混在して邪魔 |
| 同じ図を何度も載せる | 冗長 |

## 📈 CPP Modules への展開計画

cub3D で確立した原則を CPP Modules (cpp00-04) に適用する。

### 優先度高

1. **用語集** を CPP にも作る（class, OCF, 多態性 等）
2. **ASCII アート → Mermaid** に置換
3. **英語 + 日本語併記**（OCF = Orthodox Canonical Form = 正規形）
4. **ページ分割**（長いページは概念/実装/評価に分ける）
5. **「このページは何？」** セクションを全ページに追加

### CPP 固有の追加コンテンツ

| モジュール | 追加要素 |
|:---|:---|
| cpp00 | std::cout の仕組み図、class の引き出し箱比喩 |
| cpp01 | new/delete のメモリ確保アニメ GIF |
| cpp02 | OCF 4 関数の関係 Mermaid 図、演算子オーバーロードの比較表 |
| cpp03 | 継承ツリーの Mermaid 図、構築/破棄順序のシーケンス図 |
| cpp04 | 多態性の vtable 図、抽象クラスの設計図比喩 |

### CPP 向け教育 GIF 候補

| GIF | 内容 |
|:---|:---|
| new-delete.gif | メモリ確保・解放の可視化 |
| copy-deep-shallow.gif | ディープ vs シャローコピーの違い |
| vtable.gif | virtual 関数のディスパッチ |
| inheritance.gif | 構築/破棄順序のアニメーション |

- [ ] 英語 + 日本語併記

- [ ] リンクが全部繋がっている（mkdocs build --strict）

## 📌 サマリー

**cub3D ガイドが分かりやすくなった最大の理由は:**

1. 🧠 **メタファー駆動** — 懐中電灯、引き出し箱等の身近な例え
2. 📚 **用語の徹底解説** — 分からない単語をゼロに
3. 🎨 **視覚化の徹底** — Mermaid・GIF・表で文字だけに頼らない
4. ✂️ **小分けと番号付け** — 1 ページ 10〜15 分の集中力単位
5. 🔁 **英語 + 日本語併記** — カタカナ用語の意味を明示
6. 📖 **テンプレート統一** — 12 セクションで読者が構造を覚える

これを CPP Modules にも同じ粒度で適用すれば、同等の分かりやすさが実現できる。
