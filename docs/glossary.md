このページでは、cub3D で登場する用語を **全部平易な言葉で** 説明します。
分からない用語があったら、このページで確認してください。

## 🗂️ ファイル関連

### `.cub` ファイル

**cub3D 専用の「マップ設計図ファイル」** です。

テキストファイル（`.txt` と同じ中身）に、ゲームで使う **マップ情報と見た目** を書きます。

**中身の例:**

```
NO ./textures/north.xpm     ← 北向きの壁に貼る画像
SO ./textures/south.xpm     ← 南向きの壁
WE ./textures/west.xpm      ← 西向きの壁
EA ./textures/east.xpm      ← 東向きの壁
F 220,100,0                  ← 床の色（RGB）
C 100,100,255                ← 天井の色（RGB）

111111                       ← 迷路のマップ
100001                       ← 1 = 壁、0 = 通路
10N001                       ← N = 北向きプレイヤー
100001
111111
```

「**cub**」は立方体（cube）の略。3D 迷路なので立方体ブロックで構成されているイメージ。

→ 詳しくは **[02. パーサー](02-parser.md)** で `.cub` ファイルの読み込み方法を解説

## 🔧 プログラミング用語

### 構造体 (struct)

**「関連する変数をまとめて 1 つの箱にする仕組み」** です。

例えば「プレイヤー」の情報は位置 (x, y)、向きなど複数あります。
これを別々の変数にするとバラバラになるので、1 つの箱（構造体）にまとめます。

```c
// 構造体の例
typedef struct s_player {
    double x;     // X 座標
    double y;     // Y 座標
    double dir;   // 向き
} t_player;

// 使い方
t_player p;
p.x = 3.5;        // 箱の中の x に代入
p.y = 2.5;
```

イメージ: **引き出しが複数ある整理箱**

→ 詳しくは **[02. パーサー](02-parser.md)** の「構造体って何？」で `config` 構造体を実例として解説

---

### ポインタ (pointer)

**「変数の住所（アドレス）を記憶する変数」** です。

```c
int value = 42;
int *ptr = &value;  // ptr は value の住所を持つ
*ptr = 100;          // 住所経由で value を 100 に変更
// → value は 100 になる
```

イメージ: **家の場所が書かれたメモ**

→ 使用例: **[00. プログラム全体の流れ](00-main-flow.md)** で `t_game *game` の引き回しを確認

---

### 関数ポインタ

**「関数の住所を記憶する変数」** です。
状況に応じて呼び出す関数を変えるのに使います（コールバック）。

→ 実例: **[07. 入力処理](07-input.md)** の `mlx_hook` でキーイベント関数を登録する箇所

---

### コールバック (callback)

**「あとで何かが起きたら呼んでね」と先に登録しておく関数** です。

例: 「W キーが押されたら `ft_key_press` を呼んでね」と miniLibX に登録。
実際に押されたとき miniLibX がその関数を呼び出してくれます。

→ 詳しくは **[07. 入力処理](07-input.md)** のイベントループの章

---

### 二次元配列

**「配列の配列」= 格子状のデータ** です。

マップのように「縦 × 横」のデータを扱うのに便利。

```c
char map[5][8];        // 縦 5 × 横 8 のマップ
map[y][x] = '1';       // 壁を設置
```

**注意**: C では `map[y][x]` の順（縦が先、横が後）が慣習。

→ 使用例: **[02. パーサー](02-parser.md)** のマップ解析、解放は **[08. メモリ管理](08-memory.md)**

## 📐 数学・幾何関連

### ベクトル (vector)

**「方向と大きさを持つ数字の組」** です。cub3D では 2D ベクトル (x, y) を使います。

例: プレイヤーが東向き → `dir = (1, 0)`
南向き → `dir = (0, 1)`

→ 実例: **[05. カメラと魚眼補正](05-camera.md)** の `dir` / `plane` 解説

### 正規化 (normalize)

ベクトルの長さを **1 にそろえる** こと。

### FOV (Field of View)

**視野角。カメラが何度の範囲を映すか**。
cub3D は通常 **90 度** です（`|dir| = |plane|` のとき）。

→ 詳しくは **[05. カメラと魚眼補正](05-camera.md)** でカメラプレーンと合わせて解説

## 🔦 レイキャスティング用語

### レイキャスティング (raycasting) = 光線投射法

**「プレイヤーから光線を飛ばして、壁までの距離を測る」** 描画手法。
1992 年の Wolfenstein 3D で有名になりました。

→ 詳しくは **[03. レイキャスティングとは](03-raycasting.md)** で概念を、
**[04](04-dda.md)**・**[05](05-camera.md)** で実装を

### 光線 (ray)

**プレイヤーから伸ばす仮想の直線**。
実際に光を出すわけではなく、数学的に「どこまで行くと壁か」を計算します。

→ 詳しくは **[03. レイキャスティングとは](03-raycasting.md)** の「光線って何？」

### dir (direction) ベクトル

**プレイヤーの向きを表すベクトル**。`player.dir.x`, `player.dir.y` の 2 成分。

| プレイヤーの向き | dir の値（だいたい） |
|:-:|:-:|
| 北（上） | `(0, -1)` |
| 南（下） | `(0, 1)` |
| 東（右） | `(1, 0)` |
| 西（左） | `(-1, 0)` |

→ 実例: **[00. プログラム全体の流れ](00-main-flow.md)** の `ft_init_player` で方向別に設定

### plane（カメラプレーン）

**視野の横幅を表すベクトル**。`dir` に垂直な方向を指します。
`dir` と `plane` の比で **視野角（FOV）** が決まります。

→ 詳しくは **[05. カメラと魚眼補正](05-camera.md)** の「カメラプレーンって何？」

### camera_x

**画面上の x 位置を -1〜+1 の数字に変換したもの**。

| 画面の x | camera_x |
|:-:|:-:|
| 0（左端） | -1 |
| 512（中央） | 0 |
| 1024（右端） | +1 |

この数字で「どの光線か」を決めます。

→ 詳しくは **[05. カメラと魚眼補正](05-camera.md)** の「camera_x って何？」

### 1024 という数字の意味

**画面の横幅（ピクセル数）** です。`includes/cub3d.h` の `#define WIN_W 1024` で決まっています。
光線も **横幅の本数だけ** 飛ばすので、画面幅が 1024 なら **1024 本の光線** を処理します。

→ 詳しくは **[03. レイキャスティングとは](03-raycasting.md)** の「なぜ 1024 本？」

### DDA (Digital Differential Analyzer) = デジタル微分解析機

**「格子を 1 マスずつ効率よく渡る」アルゴリズム**。
壁の境界だけチェックするので、ピクセル単位より圧倒的に速い。

→ 詳しくは **[04. DDA アルゴリズム](04-dda.md)** 一本まるごと

### side

**DDA で最後にどっちの格子線を渡ったかを表すフラグ**。

| side の値 | 意味 | 当たった壁 |
|:-:|:---|:---|
| 0 | X 方向の格子線を渡った | **東 (EA) or 西 (WE) の壁** |
| 1 | Y 方向の格子線を渡った | **北 (NO) or 南 (SO) の壁** |

→ 実例: **[04. DDA アルゴリズム](04-dda.md)** のコード解説、
**[05. カメラ](05-camera.md)** の壁向き判定

### 魚眼効果 (fisheye effect)

**画面の端が膨らんで見える歪み**。
斜めの光線は長くなるので、そのまま距離として使うと起きる現象。

→ 詳しくは **[05. カメラと魚眼補正](05-camera.md)** の「魚眼効果って何？」

### perp_wall_dist（垂直壁距離）

**魚眼効果を防ぐための「まっすぐ前への距離」**。
光線の長さではなく、プレイヤーの正面方向への投影距離を使います。

→ 詳しくは **[05. カメラと魚眼補正](05-camera.md)** の垂直距離の項、
壁衝突フリーズ対策は **[09. 実際のバグと修正](09-debugging.md)**

## 💻 画面・描画関連

### WIN_W, WIN_H

画面の **幅（Width）** と **高さ（Height）**。
cub3D では `WIN_W = 1024`, `WIN_H = 768` に設定されています。

→ 定義箇所: **[00. プログラム全体の流れ](00-main-flow.md)** の init_game

### ピクセル (pixel)

**画面の最小単位の 1 点**。画面は縦横にピクセルを並べて絵を作ります。

→ 使用例: **[06. レンダリング](06-rendering.md)** の draw_column

### テクスチャ (texture)

**壁に貼る画像**。cub3D では N/S/E/W の 4 方向別に用意します。

→ 詳しくは **[06. レンダリング](06-rendering.md)** のテクスチャマッピング

### フレーム (frame)

**1 枚の絵**。ゲーム画面は 1 秒間に 60 枚の絵を切り替えて動いて見せます（60 FPS）。

→ 実例: **[00. プログラム全体の流れ](00-main-flow.md)** の `ft_game_loop`

### フレームバッファ

**画面に出す前に絵を組み立てる「下書き用キャンバス」**。
`ft_put_pixel` で 1 ピクセルずつ書き込み、完成したら一度に画面に転送します。

→ 詳しくは **[06. レンダリング](06-rendering.md)** と
**[00. プログラム全体の流れ](00-main-flow.md)** の init_game

## 🗺️ マップ関連

### マップ (map)

**迷路の設計図（二次元配列）**。`1` が壁、`0` が通路。

→ 詳しくは **[02. パーサー](02-parser.md)** のマップ解析

### マス (cell / tile)

マップの **1 マス分の区画**。座標 `(x, y)` で表現します。

### 格子線 (grid line)

**マスとマスの境界線**。DDA で「次の格子線までジャンプ」するときの基準。

→ 実例: **[04. DDA アルゴリズム](04-dda.md)**

### プレイヤー位置

**小数で持つ**のがポイント。整数なら「マス (3, 2) にいる」、小数なら「(3.5, 2.5) にいる」と中心位置まで正確に表現できます。

→ 実例: **[00. プログラム全体の流れ](00-main-flow.md)** の `ft_init_player`

### 0.5 って何？

プレイヤーを **マスの中央に置く** ための小数。
`N` を見つけたマスが `(3, 2)` なら、プレイヤー位置は `(3 + 0.5, 2 + 0.5) = (3.5, 2.5)` にセットします。

→ 詳しくは **[03. レイキャスティングとは](03-raycasting.md)** の「0.5 の意味」

## 🏗️ プログラム構造・用語

### config flags（設定フラグ）

**「`.cub` ファイルの要素を全部読んだか」を追跡するビットフラグ**。

6 つの要素（NO, SO, WE, EA, F, C）に対応する 6 ビット。

| ビット | 意味 |
|:-:|:---|
| ビット 0 | NO 読み込み済み |
| ビット 1 | SO 読み込み済み |
| ビット 2 | WE 読み込み済み |
| ビット 3 | EA 読み込み済み |
| ビット 4 | F 読み込み済み |
| ビット 5 | C 読み込み済み |

全部立つと **`0x3F`**（2 進数で `111111`）になり、「全要素揃った」と判定できます。

→ 詳しくは **[02. パーサー](02-parser.md)** の「config flags って何？」

### 状態機械 (state machine)

**「今どの状態か」を覚えながら処理を進める仕組み**。
「機械」の漢字は英語「state **machine**」の直訳で **合っています**（機械 = machine）。

cub3D では「要素読み取り中」「マップ読み取り中」の 2 状態を flags で追跡します。

→ 詳しくは **[02. パーサー](02-parser.md)** の「状態機械って何？」

## 🛠️ miniLibX 関連

### miniLibX (MLX)

**42 が提供する最小限のグラフィックライブラリ**。ウィンドウ表示、画像描画、キー入力等を提供します。

→ 初期化処理は **[00. プログラム全体の流れ](00-main-flow.md)** の `ft_init_game`

### mlx_hook

**「あるイベントが起きたら関数を呼んで」と登録する関数**。

```c
mlx_hook(win, 2, 1L<<0, ft_key_press, &game);
// 2 = キー押下イベント、ft_key_press を呼ぶ
```

→ 詳しくは **[07. 入力処理](07-input.md)**、バグ事例は **[09](09-debugging.md)**

### イベントループ (event loop)

**「何か起きたら反応する」を無限に繰り返す仕組み**。`mlx_loop()` が裏で回してくれます。

→ 詳しくは **[07. 入力処理](07-input.md)** の「イベントループって何？」

### イベントマスク

**どんなイベントを受け取りたいかを指定するビット**。
Linux 版 miniLibX（X11）では必須、macOS 版ではほぼ無視されます。

| マスク | 意味 |
|:---|:---|
| `1L << 0` | KeyPressMask（キー押下を受け取る） |
| `1L << 1` | KeyReleaseMask（キー離しを受け取る） |

→ これを間違えた実バグ事例: **[09. 実際のバグと修正](09-debugging.md)** の「バグ #1」

## 🚨 エラー・デバッグ関連

### メモリリーク (memory leak)

**`malloc` で確保したメモリを `free` し忘れて残り続けること**。
42 の評価では即不合格のフラグ。

→ 詳しくは **[08. メモリ管理](08-memory.md)** 全体

### ダングリングポインタ (dangling pointer)

**解放済みのメモリを指したままのポインタ**。アクセスするとクラッシュします。

→ 関連: **[08. メモリ管理](08-memory.md)** の cleanup 順序

### segfault (セグメンテーション違反)

**アクセスしてはいけないメモリに触ろうとしてプログラムが異常終了する現象**。
NULL ポインタを使う、配列の範囲外にアクセスする等で起きます。

→ 壁衝突時のフリーズの実例は **[09. 実際のバグと修正](09-debugging.md)**

### valgrind / leaks

**メモリリークを検出するツール**。Linux では `valgrind`、macOS では `leaks`。

→ 使い方は **[08. メモリ管理](08-memory.md)** の最後の節
