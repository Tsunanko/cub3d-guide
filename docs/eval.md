# cub3D 評価対策

---

## このプロジェクトの評価テーマ

> **レイキャスティングの仕組みを理解し、メモリリークなく実装できているか**

---

## 評価前チェック（即不合格を避ける）

以下を **1 つでも満たしていなければ、そもそも採点対象外** になる可能性あり。

- [ ] `make` がエラーも警告もなく通る
- [ ] `-Wall -Wextra -Werror` フラグが Makefile にある
- [ ] norminette が通る（NG があると即 0 点の可能性）
- [ ] **メモリリークなし** (`valgrind` / `leaks` で確認)
- [ ] 関数あたり 25 行以内、1 ファイル 5 関数以内（42 norm）

!!! danger "即不合格フラグ"

    | フラグ | 条件 |
    |--------|------|
    | **Invalid compilation** | コンパイルエラー or 警告 |
    | **Norm error** | norminette NG |
    | **Crash** | segfault / 無限ループ |
    | **Leaks** | valgrind / leaks でリーク検出 |
    | **Forbidden function** | 許可されていない関数の使用 |
    | **Can't support / explain code** | アルゴリズムを説明できない |

## Exercise 別 Q&A

### マップ解析（パーサー）

| 質問 | 模範回答 | なぜ聞かれるか |
|------|---------|---------------|
| パーサーの役割は？ | `.cub` ファイルを読んで要素（テクスチャ、色）とマップを取り出し、構造体に格納する処理 | 基本概念の確認 |
| 要素が欠けていたらどうなる？ | `config->flags` が `0x3F` にならないので "Missing element" でエラー終了 | エラーハンドリング |
| マップが壁で囲まれていないとどう判定？ | 通行可能マス（0/N/S/E/W）の上下左右に空白または端があれば不正 | 壁検証ロジック |
| プレイヤーが複数いたら？ | `player_count` でカウントして、2 以上なら "Multiple player" でエラー | バリデーション |
| 不正文字のチェックは？ | `0 1 空白 N S E W` 以外が現れたら即エラー | 入力検証 |

### レイキャスティング（最重要）

!!! danger "ここが一番聞かれる"
    **レイキャスティングのアルゴリズムを説明できないと 0 点に近い評価**。
    絶対に説明できるようにしておきましょう。

| 質問 | 模範回答 |
|------|---------|
| レイキャスティングとは？ | プレイヤーから画面の横幅分（1024 本）の光線を飛ばし、壁にぶつかった距離を測って縦 1 ピクセル幅の壁を描画する手法 |
| なぜ 3D っぽく見える？ | 距離に反比例した高さで壁を描くため。近ければ大きく、遠ければ小さく描画される |
| DDA とは？ | Digital Differential Analyzer。格子状マップを効率よく渡るアルゴリズム。X と Y の次の格子境界のうち近い方に 1 マスずつ進む |
| DDA の利点は？ | ピクセル単位で進むより圧倒的に高速。壁境界だけチェックすればよい |
| カメラプレーンとは？ | 視野角（FOV）を決める仮想平面。`dir` と `plane` の比で FOV が決まる（通常 `|dir|=|plane|` で 90 度） |
| 魚眼効果とは？ | 斜めの光線は長くなるので、そのまま距離として使うと画面端が膨らんで見える現象 |
| 魚眼補正の仕組みは？ | 実距離ではなく `perpendicular wall distance`（プレイヤーの向きに対する垂直距離）を使う |
| 壁の方向（N/S/E/W）の判定は？ | DDA で最後にどちらの格子線を渡ったか（side=0/1）と、光線の向きの符号から決定 |
| 光線が無限ループしない理由は？ | DDA は必ず 1 マスずつ前進。マップ外に出たら `break` |
| 画面 x 座標から光線方向への変換は？ | `camera_x = 2x/WIN_W - 1` で -1〜+1 に正規化し、`ray.dir = player.dir + player.plane * camera_x` |

### レンダリング

| 質問 | 模範回答 |
|------|---------|
| 壁の高さはどう決める？ | `line_h = WIN_H / perp_wall_dist`。距離に反比例 |
| テクスチャ座標 tex_x はどう計算？ | `wall_x` (壁上の位置 0〜1) に `tex->width` をかける |
| tex_x を反転する理由は？ | 光線の向きによってテクスチャが鏡像になるので補正 |
| 床と天井はなぜ単色？ | subject 要件。テクスチャ床は bonus |
| 描画の高速化は？ | `mlx_pixel_put` ではなく自前の image buffer に書き込み、一度にウィンドウに転送 |

### 入力とプレイヤー移動

| 質問 | 模範回答 |
|------|---------|
| キー入力はどう処理？ | `mlx_hook` で KeyPress/KeyRelease を登録し、フラグを立てる。毎フレーム移動関数でチェック |
| 押下と離すを分ける理由は？ | 押しっぱなしで連続移動するため。毎フレーム判定が必要 |
| 衝突判定の仕組みは？ | 次の位置を格子座標に変換し、マップ範囲内かつ `'1'` でないかチェック |
| X と Y を別々に判定する理由は？ | 壁沿いにスライド移動できるようにするため |
| 回転はどう実装？ | 2D 回転行列を `dir` と `plane` の両方に適用 |
| plane を一緒に回す理由は？ | 視野角が向きに追従する必要があるため |

### メモリ管理

| 質問 | 模範回答 |
|------|---------|
| リークをどう防いだ？ | 確保と解放を対にし、エラー時は errctx（グローバル登録）で cleanup |
| 二次元配列の解放順は？ | 内側の各行を `free` してから外側（行ポインタ配列）を `free` |
| miniLibX のリソース解放は？ | `mlx_destroy_image`、`mlx_destroy_window`、`mlx_destroy_display`（Linux のみ）+ `free(mlx)` |
| ESC 終了でもリークしない？ | `ft_cleanup` を呼んでから `exit(0)` |
| ウィンドウ × ボタンで閉じる処理は？ | `mlx_hook(win, 17, 0, ft_close, game)` で登録 |

## 概念の深掘り

### レイキャスティングの全体像

```
+-- プレイヤー視点（上から） --+
|                              |
|  plane                       |
|    +                         |
|    |                         |
|    +-------+   dir           |
|   /|       |    ↑            |
|  / |       |   /             |
| /  |       |  /              |
|/   |       | /               |
|    |   P   |/                |
|    +-------+                 |
|                              |
|   視野 (FOV)                 |
+------------------------------+

画面の各ピクセル x (0..WIN_W) について:
  1. camera_x = 2x/WIN_W - 1  (-1〜+1)
  2. ray.dir = dir + plane * camera_x
  3. DDA で格子を渡って壁にぶつかるまで進む
  4. perp_wall_dist (魚眼補正済み距離) を計算
  5. line_h = WIN_H / perp_wall_dist
  6. 壁を line_h の高さで縦線として描画
     (テクスチャから色を取って 1 ピクセルずつ)
```

### DDA の動きを図で

```
プレイヤーから →方向に光線を飛ばす

 y
 ^
 |  +---+---+---+---+
 |  | 0 | 0 | 0 | 1 |
 |  +---+---+---+---+
 |  | 0 |P→→→→→→→ 1 |
 |  +---+---+---+---+
 |  | 0 | 0 | 0 | 1 |
 |  +---+---+---+---+
 +---------------------------> x

手順:
  step1: X の次の格子線まで 0.4
  step2: Y の次の格子線まで 0.7
    → 小さい方 (X=0.4) に進む
  step3: X の次まで 1.4, Y は 0.7
    → Y に進む
  ...
  hit: map[y][x] == '1' で終了
```

## 即不合格フラグ一覧

| フラグ | 意味 |
|--------|------|
| **Crash** | 無効なマップ、配列範囲外、null デリファレンス等でクラッシュ |
| **Leaks** | valgrind / leaks でリーク検出 |
| **Norm error** | 関数 25 行超え、1 ファイル 5 関数超え、禁止構文（for, 複数変数宣言）等 |
| **Forbidden function** | 禁止関数（`printf` 等）の使用 |
| **Cheat** | 他人のコードのコピー |
| **Can't support / explain code** | **レイキャスティング / DDA を説明できない** |
| **Incomplete work** | ESC 終了や × ボタン等の基本操作が動かない |

!!! tip "ディフェンス前の 10 分チェック"
    1. `make re` が警告なく通る
    2. `valgrind ./cub3D maps/valid.cub` でリーク 0
    3. 正常マップで操作 → 違和感なし
    4. 壊れたマップ（プレイヤーなし等）で **エラー終了**
    5. ESC、× ボタンで **リークなく終了**
    6. **レイキャスティングの説明を 3 分で** 口頭で練習
